# User Stories

This document captures **technology-agnostic user stories** derived directly from the Minimal Use Case (MUC).

These stories describe **what participants want to achieve**, not how the system is implemented.

---

## 1. Identity & Participation

### US-01: Create an Identity

As a community participant, I want to create a persistent identity so that my actions and contributions can be recognized over time.

### US-02: Equal Starting Conditions

As a participant, I want everyone to start with the same base reputation so that no one has unfair initial advantage.

### US-03: Voluntary Participation

As a participant, I want to join or leave projects voluntarily so that participation is based on consent.

---

## 2. Project Proposals

### US-04: Submit a Project Proposal

As a participant, I want to submit a project proposal describing a community need so that it can be evaluated collectively.

### US-05: View All Proposals

As a participant, I want to view all active and past proposals so that I can understand community priorities.

### US-06: Transparent Proposal Details

As a participant, I want to see objectives, budgets, timelines, and required roles for each proposal so that I can make informed decisions.

---

## 3. Collective Decision Making

### US-07: Vote on Proposals

As a participant, I want to vote on project proposals so that collective priorities determine which projects are executed.

### US-08: Understand Voting Outcomes

As a participant, I want to see voting results and quorum rules so that decisions are transparent and verifiable.

---

## 4. Project Execution

### US-09: Opt-In to Execute Tasks

As a participant, I want to opt in to project execution roles so that I can contribute my labor or skills.

### US-10: Fair Selection for Execution

As a participant, I want executors to be selected among eligible volunteers using transparent rules so that favoritism is avoided.

### US-11: Track Project Progress

As a participant, I want to track milestones and task completion so that I know how the project is advancing.

---

## 5. Financial Transparency

### US-12: View Budget and Expenses

As a participant, I want to see the project budget and all expenses so that funds usage is transparent.

### US-13: Verify Payments

As a participant, I want to verify who was paid, how much, and for what work so that misuse of funds can be detected.

---

## 6. Auditing & Accountability

### US-14: Audit Project Milestones

As a participant, I want to audit project milestones so that progress claims can be verified.

### US-15: Flag Inconsistencies

As a participant, I want to flag inconsistencies or issues so that problems are addressed early.

### US-16: See Audit History

As a participant, I want to see past audits and their outcomes so that trust is based on evidence.

---

## 7. Reputation System

### US-17: Earn Reputation for Contributions

As a participant, I want to gain reputation when I contribute successfully so that good behavior is rewarded.

### US-18: Lose Reputation for Failures

As a participant, I want reputation to decrease when I fail or act dishonestly so that accountability exists.

### US-19: Reputation-Based Eligibility

As a participant, I want eligibility for execution or auditing roles to depend on reputation so that reliability matters.

---

## 8. Transparency & History

### US-20: Immutable History

As a participant, I want all actions and decisions to be recorded permanently so that history cannot be rewritten.

### US-21: Review Past Projects

As a participant, I want to review completed projects so that the community can learn from outcomes.

---

## 9. System Trust

### US-22: No Central Authority

As a participant, I want the system to operate without a central administrator so that no single actor can manipulate outcomes.

### US-23: Equal Visibility

As a participant, I want all participants to have equal visibility into system state so that trust is shared.

---

## Scope Note

These user stories define **what the system must enable**, not how it is implemented.

Algorithm design, technical architecture, and UI decisions are intentionally excluded and defined in separate documents.
